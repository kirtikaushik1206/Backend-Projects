Section 2: Deep Dive into the User Schema (models/user.js)


1. The Schema Properties
fullName & email: Required strings. email has unique: true which creates an index in MongoDB to prevent duplicate accounts.

salt: A unique, random string generated per user during signup. It is used alongside the password to prevent pre-computed table attacks (Rainbow Tables).

password: Encrypted string (never store plain passwords!).

profileImageURL: Default fallback path /images/default1.png.

role: Uses enum: ["USER", "ADMIN"] with default: "USER". It restricts values only to these two roles (prevents unauthorized role creation).

timestamps: true: Automatically adds createdAt and updatedAt fields.




2. Pre-Save Middleware (The Password Hashing Logic)
Hum direct database me plaintext password save nahi karte. Use hash karne ke liye hum use karte hain Node.js ka built-in crypto module.

📝 Code Breakdown:
JavaScript
userSchema.pre("save", async function() {
  if (!this.isModified("password")) return; // Only hash if password changed
  
  const salt = randomBytes(16).toString(); // Generate unique salt for user
  const hashedPassword = createHmac("sha256", salt)
    .update(this.password)
    .digest("hex");
    
  this.salt = salt;
  this.password = hashedPassword;
});
pre("save", ...): Isko "Mongoose Hook" bolte hain. Database me record save hone se theek pehle ye trigger hota hai.

this.isModified("password"): Agar user sirf apna naam ya avatar badal raha hai, toh hum password ko dobara hash nahi karenge (warna login kharab ho jayega).

randomBytes(16): Generates 16 bytes of cryptographically secure random data.

createHmac("sha256", salt): HMAC (Hash-based Message Authentication Code) hume normal SHA-256 se zyada security deta hai kyunki isme hash generate karne ke liye ek secret key (salt) pass ki jaati hai.


3. Custom Static Method (matchPassword)Mongoose me hum custom helper methods bana sakte hain. Humne .static method banaya jo directly Model par call ho sake (User.matchPassword(...)).📝 Code Breakdown:JavaScriptuserSchema.static("matchPassword", async function (email, password) {
  const user = await this.findOne({ email });
  if (!user) throw new Error('User not found!');

  const salt = user.salt; // 💡 IMP: Get the user's specific salt from DB!
  const hashedPassword = user.password;

  const userProvidedHash = createHmac("sha256", salt)
    .update(password)
    .digest("hex");

  if (hashedPassword !== userProvidedHash) throw new Error('Incorrect Password');
  return user;
}):
Flow: 
1. Find user by email 
2. Get stored salt 
3.Hash user-inputted password with that salt
4.Compare both hashes.

Authentication Middleware Notes
Purpose
A custom Express middleware that checks incoming HTTP requests for a specific authentication cookie. If valid, it decodes the user details from the JWT token and attaches them to req.user so subsequent routes can access the logged-in user's info.

Line-by-Line Breakdown
1. Higher-Order Middleware Wrapper (Lines 3–4)
JavaScript
function checkForAuthenticationCookie(cookieName) {
    return (req, res, next) => { ... }
}
Why this pattern? It returns an Express middleware function (req, res, next).

Flexibility: Passing cookieName allows you to reuse this middleware for any cookie key name (e.g., 'token', 'session', etc.).

2. Read Cookie (Lines 5–8)
JavaScript
const tokenCookieValue = req.cookies[cookieName];
if (!tokenCookieValue) {
    return next();
}
Extracts the token value from req.cookies. (Note: Requires cookie-parser middleware active in Express).

If no cookie exists: It calls next() immediately to let unauthenticated users proceed to public routes without throwing an error.

3. Token Verification & Payload Injection (Lines 10–14)
JavaScript
try {
    const userPayload = validateToken(tokenCookieValue);
    req.user = userPayload;
} catch(error) {}
validateToken(): Verifies the JWT signature.

req.user = userPayload: Stores user details (e.g., _id, email, role) on the req object for downstream route handlers.

try...catch: Handles invalid/expired tokens gracefully without crashing the server. If verification fails, req.user simply remains undefined.

4. Call Next Middleware (Line 15)
JavaScript
return next();
Ensures execution always moves forward to the next middleware or route handler regardless of whether authentication succeeded or failed.
